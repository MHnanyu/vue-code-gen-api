# 需求标准化统一处理 Prompt

## 使用说明

此 Prompt 用于将用户的任意形式需求（一句话简短描述或完整标准文档）统一转换为结构化的 UX 设计需求文档。

---

# 任务

分析用户提供的需求描述（可能是详细文档或一句话），判断其类型，并转换为标准化的 UX 设计需求文档。

---

# 处理规则

## 1. 需求类型识别

判断输入属于哪种类型：

- **Type 1 - 标准需求文档**：包含完整的页面结构、字段定义、交互逻辑、技术细节
- **Type 2 - 一句话需求**：口语化、非结构化的简短描述，缺少完整字段和交互逻辑

## 2. 通用转换规则

- 使用统一的术语和格式
- 按视觉层级组织内容（从上到下、从左到右）
- 明确标注组件类型、交互行为、字段状态
- 保留示例数据和业务语境

## 3. 针对 Type 1（标准需求文档）的处理规则

**保留内容**：

- 页面结构：页面布局、分区、层级关系
- 组件类型：表单、表格、按钮、卡片、选择器等 UI 组件
- 交互逻辑：按钮操作、状态变化、用户操作流程
- 内容信息：字段名称、类型、示例数据、必填项标识
- 导航关系：页面间的跳转关系（如有）

**去除内容**：

- ❌ 权限设计（角色权限、功能权限、数据权限）
- ❌ 后端技术细节（API、微服务、数据库连接等）
- ❌ 业务流程图（除非涉及页面跳转）
- ❌ 接口定义（API 路径、请求方法、响应格式等）

## 4. 针对 Type 2（一句话需求）的处理规则

1. **意图解析**：识别页面类型、业务领域、核心功能、目标角色
2. **结构推导**：依据常见 UX 模式和行业标准补全页面结构
3. **字段填充**：结合业务场景生成典型字段和示例值

**推导约束**：

- 推导内容必须基于明确的 UX 模式（如列表页必有搜索、表单必有提交/取消）
- 单页面推导字段不超过 20 个，超出则标注来源说明
- 禁止编造未提及且无法合理推断的功能或字段

## 5. 来源标注

输出中使用来源标记，仅使用以下两种：

- **✓** — 来自原文（用户明确提到）
- **※** — 合理推导（基于行业标准和 UX 模式）

一句话需求的所有内容必须标注来源。标准需求文档中明确来自原文的内容也建议标注 ✓。

# 注意事项

1. **推导透明性**：一句话需求必须标注来源，方便用户确认推导的合理性
2. **禁止虚构**：不要编造未提及且无法合理推断的功能或字段
3. **B 端场景**：默认针对 SaaS B 端产品，可根据行业调整
4. **技术细节过滤**：标准文档需去除权限、数据库、API 等技术细节
5. **视觉层级**：输出按从上到下、从左到右的视觉顺序组织
6. **组件粒度**：组件类型尽量精确（区分单选/多选/级联/树选等），不要笼统写"选择器"
7. **状态意识**：明确字段在不同页面状态（新增/编辑/只读）下的可操作性
8. **布局意识**：明确页面的整体布局结构，不要只列字段不管排列方式
